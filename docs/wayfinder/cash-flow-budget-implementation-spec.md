# Cash Flow budget implementation specification

## Scope

This specifies the Cash Flow budget vertical slice only: storage, calculation,
Tauri and web boundaries, frontend contracts, category-integrity integration,
and verification. It includes the agreed Cash Flow summary, Budget list,
detail-page, and drawer interaction boundaries.

## Settled behaviour

- A Budget has a user-chosen, non-blank name. The create form suggests a name
  derived from its selected category scope, but stores the user's value.
- Saving creates and starts the Budget immediately. There is no persisted draft
  state or persisted Budget deletion endpoint; an unsaved form is the only
  deletable draft.
- `allowance` is a non-negative integer number of minor currency units.
- Cadence is `daily`, `weekly`, `monthly`, or `yearly`. Periods are fixed local
  calendar periods: days, Monday--Sunday weeks, calendar months, and calendar
  years. Creation starts in the current period and applies the full allowance
  to that period, including matching earlier transactions in it.
- A scope contains one or more category targets. Selecting a root category
  includes its current child categories; selecting a child category includes
  only that child. Root scopes therefore update dynamically after a
  confirmed re-parenting; overlapping scopes are allowed.
- For each period, `available = allowance + carriedBalance - activity`, where
  `activity` is in-scope expenses less in-scope income. The signed available
  value carries into the next period. Active Budget values are recalculated from
  current Cash Flow data, so a past matching transaction mutation recalculates
  that period and every later carryover.
- An active Budget edit applies to its complete current period and later
  periods; it does not rewrite earlier-period configuration. A Budget can be
  deactivated or reactivated, never archived or deleted. A deactivated Budget
  remains editable and its history continues to derive from Cash Flow data, but
  is excluded from statistics and automated behavior.

## Persistence model

Add a `budgets` feature in `crates/core` and matching repository in `crates/db`.
Use UUID text identifiers and the existing `INTEGER` minor-unit convention.
Use `DATE` values serialized as `YYYY-MM-DD` for calendar boundaries and the
existing `TIMESTAMP` convention for audit fields.

### `budgets`

| Column | Notes |
| --- | --- |
| `id TEXT PRIMARY KEY` | Generated by the service. |
| `name TEXT NOT NULL` | Trimmed, non-blank; names need not be unique. |
| `cadence TEXT NOT NULL` | Checked against the four cadence values. |
| `first_period_start DATE NOT NULL` | Derived by the service from the save date and cadence. |
| `deactivated_at TIMESTAMP NULL` | `NULL` means active; non-null means deactivated. |
| `created_at`, `updated_at TIMESTAMP NOT NULL` | Match existing table conventions. |

Do not store a mutable allowance or scope on this table: both are period
configuration and must retain their prior-period values after an edit.

### `budget_revisions` and `budget_revision_scopes`

`budget_revisions` holds `id`, `budget_id`, `effective_period_start`,
`allowance`, and audit timestamps. It has a unique `(budget_id,
effective_period_start)` constraint. Creation adds the first revision. Editing
updates the revision for the current period; it never alters an earlier one.

`budget_revision_scopes` holds `revision_id` and `category_id`, with a unique
pair constraint and foreign keys to its revision and category. A scope's kind
is inferred and validated from the selected category at write time rather than
duplicated: a root target expands dynamically to its current children, while a
child target remains exact. Revisions must have at least one scope row.

The service canonicalizes the set of scope category IDs before testing the
active-Budget uniqueness rule. A second active Budget with the same cadence and
the same canonical scope set is rejected; names do not affect that comparison.

No Budget-period calculation result is persisted. Period history is derived
from the revisions and current Cash Flow data for both active and deactivated
Budgets, so later matching transaction changes continue to recalculate affected
periods and their carryover.

### Migration and indexes

Create one forward-only Diesel migration with the two tables, their foreign
keys, the cadence check, and these indexes:

- `budget_revisions (budget_id, effective_period_start)`
- `budget_revision_scopes (category_id)`
- `budgets (deactivated_at)` for active/deactivated listing

Add the tables, joinables, and multi-table allowance to `crates/db/src/schema.rs`.
Expose `budgets_repository()` from `Database`, construct `BudgetsService` in
`ServiceContext`, and keep category and Budget mutation checks in one database
transaction to avoid a category change racing a Budget update.

## Core contract

Create `crates/core/src/features/budgets/{models,service,traits}.rs` and expose
the feature from `features/mod.rs`.

```text
BudgetCadence = daily | weekly | monthly | yearly
BudgetStatus = active | deactivated

NewBudget { name, allowance, cadence, categoryIds }
BudgetUpdate { name, allowance, categoryIds }
BudgetPeriod { startDate, endDate, allowance, carriedBalance, activity, available }
Budget { id, name, cadence, status, firstPeriodStart, scope, currentPeriod?, periods? }
BudgetCategoryImpact { directlyScopedBudgets, rootScopeBudgets }
```

`create_budget` accepts no start date: it uses an injectable local-date clock so
tests can make period boundaries deterministic. `update_budget` writes only the
current-period revision for active or deactivated Budgets. `deactivate_budget`
and `reactivate_budget` are idempotent; reactivation re-checks the active
same-scope-and-cadence rule. The read service provides a list of summaries, one
Budget detail, and period history ordered newest first.

The calculator must traverse periods in chronological order because each
period's available balance is the next period's carryover. It queries only
non-deleted matching transactions in the inclusive period range and handles
daily, Monday-start weekly, month-end, and leap-year transitions explicitly.

## Category integrity

The existing transaction-category service needs an injected Budget dependency
or a coordinating Cash Flow service so it can enforce the settled rules.

- Before deleting a directly scoped category from an active Budget, return a
  `BudgetCategoryImpact` with every affected Budget. The UI must update each
  Budget's current-period scope before retrying; confirmation alone cannot
  bypass this error.
- Deleting a category included only through a root scope returns its affected
  root-scoped Budgets as a warning. A confirm flag on the existing delete
  request permits the operation after the user sees that list.
- Before re-parenting a child, return the root-scoped Budgets that would gain
  or lose it. The existing category update request gains
  `confirmBudgetScopeChange`; the update must re-check the impact atomically
  and require the flag when it is non-empty.
- Only active Budgets participate in these checks; deactivated Budgets remain
  readable and editable but do not block category mutations.

Expose a read-only preflight command so both desktop and web forms can display
the affected-Budget list before presenting confirmation.

## Tauri, web, and frontend contracts

Add thin Tauri commands in `apps/tauri/src/commands/budgets.rs`, register the
module and all commands in `apps/tauri/src/lib.rs`, and delegate directly to
`BudgetsService`:

```text
get_budgets(filters?)
get_budget(budget_id)
get_budget_periods(budget_id, page?, per_page?)
create_budget(new_budget)
update_budget(updated_budget)
deactivate_budget(budget_id)
reactivate_budget(budget_id)
get_budget_category_impact(category_id, proposed_parent_id?)
```

Extend the existing category mutation command payloads with the preflight
confirmation fields described above. Do not add `delete_budget`: a saved Budget
is active or deactivated, never a persistent draft.

Under `apps/server/src/api/cash_flow`, add a `budgets` router with matching
semantics:

```text
GET    /api/cash-flow/budgets?status=active|deactivated|all
POST   /api/cash-flow/budgets
GET    /api/cash-flow/budgets/{budget_id}
PUT    /api/cash-flow/budgets/{budget_id}
POST   /api/cash-flow/budgets/{budget_id}/deactivate
POST   /api/cash-flow/budgets/{budget_id}/reactivate
GET    /api/cash-flow/budgets/{budget_id}/periods?page=&perPage=
GET    /api/cash-flow/categories/{category_id}/budget-impact?proposedParentId=
```

Nest the router from `cash_flow/mod.rs` and add every command-to-route case to
`apps/frontend/src/commands/web-command-map.ts`. Use the established camelCase
Serde/JSON shape and the current API error envelope.

In `apps/frontend/src/features/cash-flow`, add Budget Zod schemas, inferred
types, and command wrappers using `invokeCommand` and `CommandResult`. The
allowance form schema should reuse the transaction minor-unit parsing behavior
but accept zero. The create form keeps its draft locally, suggests its initial
name from selected category labels, and sends no start date. Treat the service
response as authoritative after create, update, deactivate, reactivate, and
category-impact actions.

The main Cash Flow page shows one compact Budget summary block with total
planned activity and Budgets needing attention; it links to the dedicated
Budgets page. The Budgets page is progress-first and renders responsive
individual Budget cards. A card makes Available the primary value and includes
scope, cadence, progress, spent, and allowance. Selecting it opens a dedicated
Budget detail page, not an inline expansion or panel. That page supports
previous/next period review, the available breakdown, selected-period
transactions by default, and an All activity filter. New and edit flows use the
same bottom-drawer interaction as Categories and Transactions. Deactivated
Budgets are hidden by default behind an Include deactivated filter; detail-page
actions allow deactivation and reactivation.

## Verification criteria

- Migration tests cover a clean database and a database with existing Cash Flow
  tables; foreign keys, uniqueness constraints, and Diesel schema compilation
  all succeed.
- Core tests cover every cadence boundary, full current-period allowance,
  zero allowance, signed carryover, overspend, income, and past transaction
  create/edit/re-category/re-date/delete propagation.
- Core tests prove that an active edit affects the current period onward but not
  prior configuration, and that deactivation never freezes period derivation or
  rollover. They also prove that deactivation excludes a Budget from aggregate
  statistics and that reactivation re-checks active-scope uniqueness.
- Repository/service tests cover root expansion, exact child scopes, overlapping
  scopes, same-scope-and-cadence rejection, direct-scope deletion blocking, and
  root-scope deletion/re-parenting warnings.
- Tauri command tests and Axum handler tests cover success and domain-error
  paths. The Cash Flow command-parity test is extended with the Budget wrapper
  file and verifies every new Tauri command has a web mapping.
- Frontend tests validate Zod parsing (including zero), command arguments,
  error propagation through `Result`, and the category-impact preflight. Screen
  tests cover the compact Cash Flow summary, card-to-detail-page navigation,
  detail period and activity controls, the Budget drawer, and the deactivated
  filter with keyboard-accessible controls.
